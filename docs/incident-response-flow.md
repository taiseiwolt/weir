# Weir 障害対応フロー

**ドラフト v1.0 | 2026年3月24日**

---

## 1. 概要

本文書は、Weirプラットフォームにおける障害発生時の検知・対応・復旧・振り返りのフローを定める。1人運営（Taisei + Claude Code）を前提に、全対応を完結できる設計とする。

## 2. 障害検知方法

| 検知手段 | 頻度・タイミング | 検知対象 |
|---------|---------------|---------|
| Cowork毎時ヘルスチェック | 1時間ごと（自動） | /api/health エンドポイント、Supabase接続、Stripe接続 |
| Vercel Function Logs | 障害報告時に確認 | サーバーレス関数のエラー、タイムアウト |
| Supabase Dashboard | 障害報告時に確認 | DB接続、Edge Function エラー、RLS違反 |
| 事業者（加盟店）からの報告 | 随時 | 注文不能、ダッシュボード異常、決済エラー |
| Taisei自身による発見 | 随時 | 目視確認、テスト注文 |
| Stripe Webhook 失敗通知 | 随時（Stripeから自動） | Webhook配信失敗 |

## 3. Severityレベル定義

| Severity | 定義 | 対応開始目標 | 復旧目標 | 例 |
|----------|------|-----------|---------|---|
| **Critical** | サービス全体が停止、または決済・個人情報に影響 | 15分以内 | 2時間以内 | DB接続不能、Stripe決済全面停止、個人情報漏洩 |
| **High** | 主要機能の一部が停止、複数加盟店に影響 | 1時間以内 | 4時間以内 | 特定のEdge Functionエラー、注文処理の一部失敗、ダッシュボード表示不能 |
| **Medium** | 一部機能に不具合、影響は限定的 | 4時間以内 | 24時間以内 | 特定ブラウザでのレイアウト崩れ、ポイント計算の不整合、画像表示エラー |
| **Low** | 軽微な表示問題、ユーザー体験への影響が小さい | 24時間以内 | 72時間以内 | UIの軽微なバグ、誤字、非クリティカルなログエラー |

## 4. 障害切り分けフロー

障害を認知した場合、以下の順序で原因を切り分ける。

### Step 1: Vercel（フロントエンド・API）

```
確認項目:
□ Vercel Status Page (https://www.vercel-status.com/) に障害報告があるか
□ /api/health の応答ステータスとレスポンスタイム
□ Vercel Function Logs にエラーが出ていないか
□ vercel.json の設定に変更がないか

切り分け結果:
→ Vercel側の障害: Vercel復旧を待つ + 事業者に通知
→ API関数のエラー: Step 5 へ（復旧対応）
→ 問題なし: Step 2 へ
```

### Step 2: Supabase（DB・Edge Functions）

```
確認項目:
□ Supabase Dashboard でDB接続状態を確認
□ Edge Functions のログにエラーがないか
□ RLS ポリシーの変更がないか
□ マイグレーション実行後であれば、直近のマイグレーション内容を確認

切り分け結果:
→ Supabase側の障害: Supabase復旧を待つ + 事業者に通知
→ Edge Function エラー: Step 5 へ（復旧対応）
→ DB/RLS 問題: Step 5 へ（復旧対応）
→ 問題なし: Step 3 へ
```

### Step 3: Stripe（決済）

```
確認項目:
□ Stripe Status Page (https://status.stripe.com/) に障害報告があるか
□ Stripe Dashboard で直近のWebhookイベントの配信状況を確認
□ Payment Intent の作成・確定が正常に行われているか
□ Connect アカウントの状態に問題がないか

切り分け結果:
→ Stripe側の障害: Stripe復旧を待つ + 事業者に通知（決済関連のみ影響）
→ Webhook 配信失敗: Webhookエンドポイントの修正
→ 問題なし: Step 4 へ
```

### Step 4: 外部API（LINE、Claude API、OpenAI API）

```
確認項目:
□ LINE Messaging API のステータス
□ Claude API / OpenAI API の応答
□ Google Places API / Reviews API の応答
□ X API / Instagram API の応答

切り分け結果:
→ 外部API障害: 該当機能のみ一時停止 + 事業者に通知（サービス全体には影響しない）
→ APIキー期限切れ/無効: キーの更新（docs/api-key-rotation-guide.md 参照）
→ 問題なし: アプリケーションコードの問題として Step 5 へ
```

### Step 5: 復旧対応

原因が特定できた場合、以下の手順で復旧を行う。

## 5. 復旧対応の手順

### 5-1. コード修正が必要な場合

```
1. 現象の再現確認（本番環境またはローカル環境）
2. 原因コードの特定
3. Claude Code で修正を実施
4. git commit（メッセージ: "fix: {障害内容の要約}"）
5. vercel --prod でデプロイ
6. デプロイ後に /api/health で動作確認
7. 障害が解消されたことを確認
```

### 5-2. Edge Function の修正が必要な場合

```
1. Supabase Dashboard でログを確認
2. Claude Code で修正を実施
3. supabase functions deploy {function-name} でデプロイ
4. テスト注文等で動作確認
```

### 5-3. DB修正が必要な場合

```
1. 影響範囲を SELECT で確認（変更前に必ず実施）
2. 修正SQLを作成
3. マイグレーションファイルとして保存（supabase/migrations/）
4. SQL実行
5. 実行後の状態を SELECT で確認
```

### 5-4. 環境変数・設定の修正が必要な場合

```
1. 変更前の値を記録
2. Vercel Dashboard / Supabase Dashboard で値を更新
3. 必要に応じて再デプロイ
4. 動作確認
```

## 7. エスカレーション基準

1人運営のため、エスカレーションは外部リソースへの依頼を意味する。

| 条件 | エスカレーション先 | 対応 |
|------|----------------|------|
| Supabase基盤の障害で自力復旧不能 | Supabase サポート | サポートチケット起票 |
| Stripe決済の障害で自力復旧不能 | Stripe サポート | サポートチケット起票 |
| 個人情報漏洩の疑い | 外部弁護士 | 法的対応の相談、個人情報保護委員会への報告判断 |
| Vercel基盤の障害で自力復旧不能 | Vercel サポート | サポートチケット起票 |
| セキュリティインシデント（不正アクセス等） | 外部セキュリティ専門家 | 調査・フォレンジック依頼 |

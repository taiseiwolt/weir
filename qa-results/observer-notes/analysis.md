## テスト漏れ指摘

### 1. [重大] 注文一覧APIの認可バイパス（権限チェック不足）
- **場所**: `api/orders/[...path].js` L424-455 `handleList()`
- **問題**: 未認証ユーザーが `store_id` パラメータを指定すると、その店舗の全注文一覧が取得できる。認証チェックが不在で、store_idを知っていれば誰でもアクセス可能
- **該当テスト項目**: B-09/C-06相当だが、API直接アクセスでの認可バイパスは47項目に含まれていない

### 2. [重大] 注文詳細APIのスタッフ権限チェックが未実装
- **場所**: `api/orders/[...path].js` L485 `const isStoreStaff = true; // TODO: implement store staff check`
- **問題**: 認証済みユーザーであれば、注文IDを知っていれば他店舗の注文詳細も閲覧可能（isStoreStaff が常にtrue）
- **該当テスト項目**: なし（認可テストとしてカバーすべき）

### 3. [中] Edge Function の CORS が `*` ワイルドカード
- **場所**: `supabase/functions/stripe-create-payment-intent/index.ts` L24-27
- **問題**: `Access-Control-Allow-Origin: *` により、任意のオリジンからEdge Functionを呼び出し可能。決済関連のエンドポイントでは特にリスクが高い
- **該当テスト項目**: D-01〜D-04（セキュリティ系）でカバーすべきだが明示的に含まれていない可能性

### 4. [中] innerHTML によるXSSリスク（weir-order-store.html）
- **場所**: `weir-order-store.html` 複数箇所（L1548, L1588, L1599, L1691, L1774, L1790等）
- **問題**: `innerHTML` で動的コンテンツを挿入している箇所が多数。商品名やオプション名にスクリプトが含まれている場合、XSSが成立する可能性がある。DB側でサニタイズされている前提だが、テスト項目でXSSチェックが明示されていない
- **該当テスト項目**: A系列のUI テストで網羅すべき

### 5. [中] 注文キャンセルAPIで店舗権限チェックなし
- **場所**: `api/orders/[...path].js` L504-519 `handleCancel()`
- **問題**: `requireAuth()` で認証済みであることのみ確認。認証されたユーザーであれば、他店舗の注文もキャンセル可能

### 6. [低] idempotency_key がセッション固定

- **問題**: sessionStorageに保存された冪等性キーは、同一セッションで2回目の注文時にも再利用される。注文成功後にクリアされない場合、2回目の注文が拒否される可能性がある
- **該当テスト項目**: A-06〜A-13 の注文フローテスト内でカバーすべき

### 7. [低] Webhook でのorder更新にRLS考慮なし

## 暗黙の前提で検証すべき事項

1. **手数料計算の精度**: CLAUDE.mdに「手数料は割引前の合計金額に対して計算」とあるが、Edge Function（`stripe-create-payment-intent`）では割引後の `totalAmount` に対して `feeRate` を適用している（L208-209）。ビジネスルールとの不一致の可能性
2. **capture_method の不整合**: Vercel API側（`api/orders`）は `capture_method: 'manual'`（authorize-on-order）だが、Edge Function側は `capture_method` を指定していない（デフォルト = automatic）。「authorize-on-order → capture-on-delivery」のビジネスルールと不一致
3. **order_type の表記揺れ**: Edge Function では `takeout` を `pickup` に変換しているが、Vercel API側では `takeout` のまま処理。DBに格納される値が経路によって異なる可能性
4. **ゲストPII保護**: CLAUDE.mdに「ゲストPII（名前・メール）は事業者に共有しない」とあるが、ordersテーブルの `customer_name`, `customer_email` に直接格納されており、ダッシュボードで事業者に表示される可能性
5. **サービス料10%の根拠**: Edge Functionで `serviceChargeRate = 0.10` がハードコードされているが、店舗ごとの設定や注文タイプごとの分岐がない。全注文一律10%が正しいビジネスルールか要確認
6. **ポイント残高の競合制御**: `handleConfirm()` でポイント消費時に `balance_after` を計算しているが、同時注文で残高がマイナスになるケースの排他制御がない
